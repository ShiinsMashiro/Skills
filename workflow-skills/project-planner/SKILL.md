# Project Planner (项目规划器)

**遵循 s03 + 文章优化：没有计划就走到哪算哪**

> *"给AI的信息越充分，产出越好"*

## 核心理念（来自文章）

### 1. 计划先行 (Plan First)
- 收到任务 → 先分析需求
- **信息越充分 → 产出越好**
- 复杂任务 → 先写PRD或Spec

### 2. 文档主导 vs 原型主导

```
📄 文档主导 (大功能):
  需求发散 → 写PRD → Plan+Review → 实现 → 测试迭代 → 人工Review

🎨 原型主导 (小功能/需求模糊):
  快速原型 → 验证需求 → 切换文档驱动
```

### 3. 关注架构而非细节
- 代码主干 ✓
- 系统交互点 ✓
- 语法细节 ✗

---

## 智能工作流选择

| 任务类型 | 工作流 |
|----------|--------|
| 大功能/方向明确 | 文档主导 |
| 需求模糊/小功能 | 原型主导 |
| Bug修复 | 原型主导 |
| 新系统架构 | 文档主导 |

## s03 核心：TodoWrite + 步骤规划

```
用户: "帮我做一个股票网站"

规划流程:
1. 理解需求 (1分钟) ← 信息收集
2. 列出步骤 (TodoWrite)
3. 按步骤执行
4. 每步确认
```

**完成率对比**:
- ❌ 无规划: ~30% 完成
- ✅ 有规划: ~80% 完成

---

## 规划工作流

### Step 1: 理解需求 + 信息收集 (Plan First)

```
# 收集充分信息
- 项目背景
- 现有代码结构
- 约束条件
- 预期成果
```

问关键问题：
- 目标是什么？
- 受众是谁？
- 有什么约束？
- 预期成果？

### Step 2: 列出步骤 (TodoWrite)

```markdown
## 📋 任务清单

- [ ] 步骤1: 需求分析
- [ ] 步骤2: 技术选型
- [ ] 步骤3: 架构设计
- [ ] 步骤4: 实现
- [ ] 步骤5: 测试
```

### Step 3: 按步骤执行

```
for step in todo_list:
    execute(step)
    if need_review:
        ask_user_confirmation()
    if test_needed:
        auto_generate_tests()
```

### Step 4: 关注架构

```
# 实现时关注
- 模块划分 ✓
- 接口设计 ✓
- 系统交互 ✓
- 变量命名 ✗
- 代码格式 ✗
```

1. **Plan First**: 收集充分信息
2. **TodoWrite**: 步骤可视化
3. **智能选择**: 文档vs原型
4. **架构优先**: 关注系统交互
